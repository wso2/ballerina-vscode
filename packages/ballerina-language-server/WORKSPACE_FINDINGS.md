## Candidate Memory Lessons

- Language-server test modules that create many `BallerinaLanguageServer` instances need the unified workspace facade to be `AutoCloseable`; otherwise event-bus/compiler resources and cached projects can accumulate until later tests OOM.
- Cached build projects can be stale immediately after a new `.bal` file is physically created. If the path exists on disk but `project.documentId(path)` cannot resolve it from the cached project, reload/refresh the project before serving document reads.
- File-generation utilities should not rely on `WorkspaceManager.loadProject(...)` throwing `ProjectException` to mean "file missing"; the unified facade can wrap loader failures differently. Check filesystem existence before load when the caller owns file creation.
- Non-file URI-scoped workspace views should keep descendant paths in the same overlay when the original request URI identifies a directory/project root; expression-editor contexts may resolve that request to a concrete fallback `.bal` document before semantic reads.
- Focused LS-extension failures can be caused by stale upstream module jars; if a failure disappears after recompiling an upstream module and there is no retained source diff, record the rebuild evidence before changing behavior.
- Project-root semantic-model reads should resolve the module through `ProjectService.module(...)` instead of deriving a `DocumentId` first; project-root callers need the default module's semantic model for module-symbol queries.
- If a focused LS-extension class fails but the named full-log config cases pass, run the isolated failing method and rerun the class before changing behavior; expression-editor tests can expose transient shared-state or scheduling effects that clear without a source diff.
- Full `flow-model-generator-ls-extension` completion failures that report zero completions can be downstream symptoms of an earlier `ballerina:http` compiler-plugin initialization failure (`ctxData is null`); focused completion runs may pass because they do not reproduce that suite-level compiler state.
- The `ballerina:http` `ctxData is null` compiler-plugin fault likely requires a shared guard across all direct `Package.getCompilation()` callers that can share the compiler plugin cache. CE-only serialization is insufficient because flow-model request handlers still compile directly through `WorkspaceManagerFacadeImpl.semanticModel(...)`, and model-generator utilities also use their own per-project lock in `PackageUtil`.
- A JVM-wide guard around direct `Package.getCompilation()` calls can eliminate the observed `ballerina:http` `ctxData is null` stack trace from full `flow-model-generator-ls-extension` logs, but it does not by itself fix `DataMappingCompletionTest.test[2](proj3.json)` returning zero completions in the full suite.
- Fire-and-forget language-server shutdown in tests can leak cross-class workspace/LS teardown into the next TestNG class. Waiting for the `shutdown` request response fixed one ordered zero-completion reproducer, but additional completion-path ordering races remain.
- The remaining `DataMappingCompletionTest.test[2](proj3.json)` zero-completion failure is reproducible with the ordered expression/config/data-mapping slice and disappears in the data-mapping-only block. Treat it as a completion request lifecycle/state problem before changing data-mapping completion fixtures.
- URI-scoped workspace views must load scoped `project(Path)` requests through the preserved non-file URI, not only route `document`/`module`/`semanticModel`; completion code may ask for the current project while resolving visible modules.
- Debouncing should cancel only pending expression-editor requests. Interrupting already-running completion requests can surface as swallowed `CancellationException` or empty completion results.
- Shared LS-extension test fixture state should be instance-scoped. Static `sourceDir`/`configDir` fields in a base TestNG class can cross-contaminate multi-class ordered slices even when focused class runs pass.
- `test-manager-service-ls-extension` can restore cached test outputs for the exact module command. Use `cleanTest` plus `--no-build-cache` when a fresh execution timing is needed; a one-off `GetFunctionModelTest` symbol-resolution failure did not reproduce in the focused class or fresh full rerun.
